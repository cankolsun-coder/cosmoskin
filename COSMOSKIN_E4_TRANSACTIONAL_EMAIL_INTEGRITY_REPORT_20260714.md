# COSMOSKIN E4 — Transactional Email, Invoice, Refund, Product Image & Brand Template Integrity
**Date:** 2026-07-14 · **Scope:** email templates, dispatch idempotency, image safety, brand shell, preview/test tooling. No checkout pricing, refund calculation, stock accounting, coupon or fiscal-provider logic touched. No emails sent to real customers. No deploy.

## 1. Refund completed email
**Before:** wired in [functions/api/admin/refunds.js] (send on `status=completed` after the refund insert) and a *second* trigger in [functions/api/admin/returns.js] (`status=refunded`) — the two paths could double-email the same refund; the template showed only order totals (no refund amount/type/reference/payment method); dedup existed only for return-linked refunds at the record level.
**After:** both paths dispatch through `sendRefundCompletedEmailOnce` ([functions/api/_lib/lifecycle-emails.js]) behind a **durable two-layer claim acquired before any send** (exactly-once, not read-before-send): **Layer A** — compare-and-set on the persisted `refund_records.metadata.email` (`(none|failed) → pending_claimed → sent|failed`, conditional update verified to claim exactly one row via the new `updateRowsWhere`); **Layer B** — a `pending` `email_events` claim row whose `metadata.idempotency_key` is unique-enforced by the migration's partial index, arbitrating the record-less returns path. The key converges across endpoints: `refund_completed:return:{return_request_id}:{email}` for return-linked refunds (both refunds.js and returns.js derive it), `refund_completed:refund:{refund_id}:{email}` for standalone refunds. Duplicate triggers — including the repeated admin completion callback, now routed through the dispatcher — return `skipped_duplicate`. A claim-store outage **refuses to send** (`refund_email_claim_unavailable`, retryable) instead of failing open. A stale `pending_claimed` (>15 min, `REFUND_CLAIM_STALE_MS`) is abandoned-claimant takeover territory; fresh claims are never stolen. The email renders an **İade Bilgisi** block — amount, full/partial (*Tam iade / Kısmi iade*), provider reference, payment method — from the already-persisted `refund_records` row (paid snapshot values; **no recalculation**), plus the bank-reflection-time disclaimer. Email failure settles both layers to a retryable `failed` state (`pending_retry: true`) and can never roll back the committed refund; a successful retry sends exactly once. Legitimate multiple partial refunds each have their own refund id → each correctly gets its own email (a second completed refund for the same return is already blocked at the record level).

## 2. Invoice-ready email
**Before:** did not exist in any form (no copy, no event type, no trigger — invoice_records were silently created/updated by admin).
**After:** `maybeSendInvoiceReadyEmail` sends **only** when the invoice row exists, `invoice_status ∈ {issued, ready}`, and an HTTPS PDF URL is present; recipient comes from the owning order. Idempotency is two-layer: primary stamp in `invoice_records.metadata.email` (`version_key` derived from invoice_number+pdf_url — a re-issued PDF is a new version and may legitimately re-email; the same version never does), secondary via `email_events`. Naming drift (`invoice_status|status`, `pdf_url|file_url|invoice_pdf_url`, `invoice_number|number`) is normalized by `normalizeInvoiceForEmail` **without renaming DB columns**. Hooked into admin invoice create + update ([functions/api/admin/invoices.js]); dispatch never blocks the admin response. Content: name, order no, invoice no, issue date, total, **“Faturanı Görüntüle”** CTA → PDF URL + secondary “Hesabımdaki Faturalar”. An empty invoice row (no PDF / pending) is a logged skip (`invoice_pdf_missing` / `invoice_not_ready`), never a send.
**DB note:** `invoice_ready` (and `order_cancelled`, below) require the additive CHECK-constraint migration `supabase/migrations/20260714161845_e4_email_event_types.sql` (idempotent constraint swap, same pattern as 20260511/20260628/20260702; **not executed** — manual apply per runbook). Until applied, sends work and dedup holds via the invoice metadata stamp; only the event-log insert fails visibly in logs.

## 4. Product-image root cause
Emails resolved images via per-file `absoluteUrl()` helpers with **no validation**: relative paths depended on `PUBLIC_SITE_URL` being correct (a `pages.dev`/localhost env value would silently ship broken/preview URLs into inboxes), empty catalog lookups produced imageless rows, `http:` passed through, and two files carried diverging override maps (v3 vs v4 of the same asset). Fix: one canonical `resolveEmailProductImage(productOrOrderItem, env)` in [functions/api/_lib/email-brand.js] — absolute **HTTPS** on the canonical origin, `http→https` upgrade, rejects localhost/127/file/data/blob/pages.dev and any exotic protocol, supports `email_image|image|image_url|product_image|thumbnail|metadata.image|product.image`, and falls back to the hosted branded mark so an email can never render an empty/undefined `src`. `getEmailOrigin` also fails closed to `https://www.cosmoskin.com.tr` if env is misconfigured. All templates render images through `emailImageTag`/`emailProductThumb` (explicit width/height, `display:block`, `border="0"`, alt, fallback background, no lazy-loading, no CSS background-image for product content).

## 8. Event idempotency
Keys: `refund_completed:{return:{return_request_id}|refund:{refund_id}}:{email}` and `invoice_ready:{invoice_id}:{version}:{email}` stored in `email_events.metadata.idempotency_key`. **Refunds are claim-based, not lookup-based:** `claimEmailEvent()` inserts a `pending` claim row *before* the send; the migration's partial unique index (`email_events_idempotency_claim_uniq`, over `metadata->>'idempotency_key'` where status ∈ pending/sent) guarantees only one concurrent caller can hold it, and `settleEmailEventClaim()` CAS-settles `pending → sent|failed|skipped`. `failed`/`skipped` rows leave the index so a failed send stays retryable while `sent` blocks forever. The refund record itself carries the primary claim/stamp (`refund_records.metadata.email`); invoice keeps its metadata-stamp primary layer, with `hasSentEmailEvent()` (fail-open, best-effort) demoted to a secondary check everywhere. Statuses used: `pending` (live claim) / `sent` / `failed` (+`pending_retry` metadata) / `skipped` (+`reason: skipped_duplicate`) — all allowed by the DB status CHECK. Send failures are recorded, never swallowed; claim-store outages refuse to send rather than risk duplicates.
