# COSMOSKIN — B2: Bank Transfer Rejection / Cancellation Finalization Plan

**Date:** 2026-07-05
**Type:** Planning document only. No files modified, no migrations created, no SQL run, nothing deployed.
**Scope:** Make admin rejection/cancellation of a Havale/EFT ("bank transfer") order produce a consistent, fully-finalized failed/cancelled commerce state — symmetric to what B1 already did for approval — without altering the approval path, the iyzico callback path, checkout, or any other unrelated logic.
**Relationship to B1:** B1 (`COSMOSKIN_B1_BANK_TRANSFER_FINALIZATION_PLAN_20260705.md` / `..._REPORT_20260705.md`) fixed the approval side and explicitly deferred the rejection side (its §7). This plan closes that deferred gap. Where useful, this plan reuses B1's helper file, idempotency pattern, and validator/test conventions rather than inventing new ones.

## 3. Canonical helper strategy — recommendation

**Recommended: Option A — add `rejectManualBankTransferPayment()` to the existing `functions/api/_lib/commerce-finalization.js`.**

Rationale for rejecting the alternatives (mirrors B1's own reasoning in kind):
- **Option B (separate helper file)** would split the "manual bank-transfer confirmation" concern across two files for no functional benefit — `confirmManualBankTransferPayment()` and the proposed `rejectManualBankTransferPayment()` are two faces of exactly one concern (manual bank-transfer payment-outcome finalization) and share the same order-loading, `payment_method` guard, and `payment_events`-based idempotency pattern almost line-for-line. Keeping them side by side in one file makes the symmetry (and any future audit of it) trivial to see and enforce.
- **Option C (patch the existing admin routes only, no shared helper)** would mean writing the `payments`/`payment_events` logic twice (once for `admin/orders.js`, once for `admin/orders/[id]/status.js`), which directly risks the exact "two divergent code paths with different side effects" problem the P0/P1 plan already flagged as B3/P1-3 — and would make it easy for the two routes' rejection behavior to drift apart the same way their approval behavior had already drifted apart before B1.
- **Option A** gets full de-duplication with the smallest possible blast radius, in the one file already designated for exactly this purpose, following the precedent B1 already established and validated.

Concretely:

1. **New function in the existing `functions/api/_lib/commerce-finalization.js`**, placed immediately after `confirmManualBankTransferPayment()`:

   ```
   export async function rejectManualBankTransferPayment(context, orderId, { rejectedByEmail = null, rejectedByAdminId = null, reason = null } = {})
   ```

2. **Behavior, mirroring `confirmManualBankTransferPayment()`'s shape exactly:**
   - Load the order (`id,payment_method,payment_status,status,fulfillment_status,order_number`, same select as the approval helper).
   - **Guard: not a bank-transfer order** → throw, identical in spirit to the approval helper's guard (never reachable for a card order; can never write a conflicting `payment_events` row over the iyzico-owned audit trail).
   - **Guard: already paid/settled** → this is the new "already-paid protection" requirement (§6). If `order.payment_status === 'paid'` or `order.status` is in `('paid','preparing','shipped','delivered','refunded','partially_refunded')`, return `{ ok: false, idempotent: false, blocked: true, reason: 'already_paid_or_settled' }` **immediately, before any write** — this must never release converted inventory, never release a used coupon, and never write a `payment_events` rejection row over a real, completed sale. This check lives *inside* the helper (not only in the calling routes' own pre-existing 409 guards) so the protection holds even if a future caller forgets to check first — defense in depth, not a replacement for the existing route-level guards, which stay exactly as they are.
   - **Idempotency gate, before any write**: select `payment_events` for `order_id=orderId, provider='bank_transfer', event_type='bank_transfer_payment_rejected', status='processed'`. If found, return `{ ok: true, idempotent: true, reason: 'already_rejected' }` immediately — no `payments` write, no inventory call, no coupon-release call, no email trigger from this function.
   - If neither guard fires: find the latest `payments` row for the order (`provider='bank_transfer'`); if it exists and is not already `'failed'`/`'cancelled'`, update it to **`status: 'failed'`** (chosen over the also-valid `'cancelled'` literal specifically to mirror `orders.payment_status`'s own value for this same event — both `'failed'` and `'cancelled'` pass `payments_status_final_chk`, but using the same word as `orders.payment_status` keeps the two tables' reporting/reconciliation vocabulary aligned; see §4), stamping a small `raw_callback_response`/metadata object (`source, rejected_by_email, rejected_by_admin_id, reason, confirmed_at` — reusing the exact column B1 already writes to) — same defensive-only pattern as the approval helper's own `payments` update.
   - Defensive `orders` fallback update, same spirit as the approval helper's own fallback block (a no-op in the primary call site, since the generic block in `admin/orders.js`/`status.js` already sets these fields first): if `order.payment_status !== 'failed'`, set `payment_status:'failed'`, `status:'cancelled'` (unless already a terminal/settled status), `fulfillment_status:'cancelled'`.
   - Calls `releaseInventoryReservations(context, orderId, 'admin_bank_transfer_rejected')` (imported from the existing, unmodified `functions/api/_lib/inventory.js` — no change to that file). Safe to call even though the caller may already have called it once, exactly like the approval helper's redundant-but-safe `convertInventoryReservations()` call.
   - Releases the coupon: if `order.coupon_code` is set, update the matching `coupon_redemptions` row (`order_id`, `code`) to `status:'released'` **only if its current status is not already `'released'`** (new idempotency pre-check, closing the partial gap noted in §2) — this is the same inline logic already in `admin/orders.js` today, relocated and guarded, not rewritten.
   - **Does not call `finalizeCommerceAfterPayment()` and does not call `ensureShipmentShell()`** — this is the single most important structural difference from the approval helper, and must be enforced by the validator (§9): a rejected order must never gain a coupon `'used'` status, an invoice shell, a loyalty award, or a shipment shell. This directly satisfies §4's "no invoice shell created if payment never completed" and "no loyalty points awarded" requirements by construction (by simply never calling the functions that would do either).
   - Inserts the `payment_events` row that makes the idempotency gate above work on the next call: `{order_id, provider:'bank_transfer', provider_payment_id:null, event_type:'bank_transfer_payment_rejected', status:'processed', raw_reference:null, processed_at:now(), metadata:{source:'admin_manual_bank_transfer_rejection', rejected_by_email, rejected_by_admin_id, order_number, reason, inventory_release, coupon_release}}`. No CHECK constraint exists on `payment_events.provider`/`event_type`/`status` (confirmed by B1, unchanged) — no migration needed for this row.
   - Returns `{ok:true, idempotent:false, inventory_release, coupon_release}`.

3. **`functions/api/admin/orders.js` call site** — inside the existing `if (body.action === 'mark_bank_transfer_not_received')` block (lines 357-362): call the new helper (guarded, as today, on this specific action only — **not** on generic `cancel_order`, to keep this pass's blast radius identical to what the dedicated rejection button already does; see §11 for why extending coverage to `cancel_order` is called out as a *separate*, optional decision rather than bundled silently into this fix), passing `rejectedByEmail: getAccessEmail(context)`, `reason: body.message || null`. The existing inline `releaseInventoryReservations()`/`coupon_redemptions` update in this block can either be left as-is (now redundant but harmless, exactly like B1 left the generic block's own `convertInventoryReservations()` call in place alongside the new helper's internal call) or removed in favor of the helper's own internal calls — **recommend removing the two now-duplicated lines** or keep them but only if the redundant call remaining in the historically byte-diffed portion doesn't confuse a future auditor; this exact choice should be confirmed with the user before implementation, since either is behaviorally safe.
4. **Email de-dup fix (bundled into the same change)**: change the email-send condition at line 361 from unconditional to also require `before.status !== 'cancelled' && rejectionConfirmation?.idempotent !== true` — mirroring the exact pattern B1 used for the `payment_confirmed_manual` email (item 4 of B1's plan §4), and explicitly **not** relying on querying `email_events` for a prior send (per the discovered bug in §1.5 — a log-based check would be unreliable today regardless of this fix).
5. **`functions/api/admin/orders/[id]/status.js` call site** — inside the existing `if (body.status === 'cancelled')` branch (lines 41-44): add the identical `if (current.payment_method === 'bank_transfer') await rejectManualBankTransferPayment(...)` call, mirroring exactly how B1 wired this same file's `paid` branch. Additionally fix the pre-existing, unrelated-to-bank-transfer bug noted in §1.3: add `created_by: getAccessEmail(context) || 'admin'` to this route's `order_status_events` insert (line 51) — this one-line fix applies to **every** status transition through this route, not only cancellation, exactly as B1's equivalent fix to `admin/orders.js`'s `recordEvent()` call applied to every action in that file, not only `mark_payment_paid`.

**Design properties satisfied:** avoids duplicating rejection logic across two route files (Option A benefit); idempotent via a single `payment_events` gate, exactly mirroring B1; defense-in-depth already-paid guard lives inside the helper, not only in the callers; never touches coupon/invoice/loyalty in a way that would create a `'used'` coupon, an invoice shell, or a loyalty award for a payment that never completed; never changes `finalizeCommerceAfterPayment()`, `ensureShipmentShell()`, `confirmManualBankTransferPayment()`, or any B1 approval behavior (all untouched, verified by the validator in §9); never changes iyzico callback behavior (not touched at all by this plan).

## 5. Idempotency plan

| Scenario | Behavior | Mechanism |
|---|---|---|
| Repeated rejection click (same order, action resubmitted) | Second call returns `{ok:true, idempotent:true}` from the helper; no second `payments` write, no second `payment_events` row, no second coupon-release write, no second email. | `payment_events` gate (new), checked first, before any write — identical pattern to B1. |
| Rejecting an already-cancelled order | Same as above — the order's own `status`/`payment_status` are already at their terminal values, and the `payment_events` gate short-circuits before any write is attempted. Distinct from "already-paid" (§6): this is a safe no-op, not a blocked error. | Same `payment_events` gate. |
| Rejecting an already-paid order | **Blocked**, not idempotent — returns `{ok:false, blocked:true, reason:'already_paid_or_settled'}` from the helper, in addition to the existing, unchanged, external 409 guard in both calling routes (`admin/orders.js` line 335, `admin/orders/[id]/status.js` line 38) that should prevent this call from ever being reached in the first place. See §6 for the full defense-in-depth rationale. | New internal guard inside `rejectManualBankTransferPayment()`, checked before the idempotency gate (an already-paid order can never have a prior rejection event to be idempotent against, so checking payment status first is both correct and cheaper). |
| Rejecting after approval (order was already confirmed paid via B1's helper) | Same as "already-paid" above — `confirmManualBankTransferPayment()`'s own `payment_events` row (`bank_transfer_payment_confirmed`) has no bearing on this check; the guard here is purely on the order's own `payment_status`/`status`, so it correctly blocks regardless of *how* the order became paid. | Same internal guard. |
| Cancelling after rejection (generic `cancel_order` or a second `mark_bank_transfer_not_received` on an already-rejected order) | Safe no-op via the `payment_events` gate, same as "already-cancelled" above. If `cancel_order` (not the dedicated action) is the one resubmitted, per §3 item 3's scoping decision, the new helper is not called for that action at all in this pass — so this specific combination produces no new side effect either way, only the pre-existing generic order/event update, unchanged from today. | `payment_events` gate (for the dedicated action); scoping decision (for the generic action). |
| Rejection email de-dup | Gated on `before.status !== 'cancelled' && rejectionConfirmation?.idempotent !== true` — the first condition covers the common case (repeat click reads an already-`'cancelled'` `before` snapshot); the second guards the same narrow concurrent-double-click race B1 also documented and accepted for its own email gate. | New (mirrors B1's `payment_confirmed_manual` gate exactly). |
| `payment_events` de-dup | The gate itself. | New, as above. |
| Inventory release idempotency | Already achieved today — `release_order_inventory` only matches `status='reserved'` rows; a second call (from the helper, redundant with the generic block's own call) matches zero rows and no-ops. Confirmed, not assumed (same RPC family B1 already verified for `convert_order_inventory`). | Existing, unchanged. |
| Coupon release idempotency | **New pre-check added** — the helper only issues the `coupon_redemptions` update if the row's current `status` is not already `'released'`, closing the "no idempotency pre-check" partial gap from §2. Functionally harmless either way (repeating the same `UPDATE ... SET status='released'` is not incorrect), but the explicit check keeps this helper's shape consistent with every other idempotency-checked side effect in both this helper and B1's, and avoids an unnecessary write + `updated_at` bump on a no-op repeat call. | New. |

**Residual race window (documented, consistent with B1's own §5):** the same plain JS check-then-act race B1 accepted for its approval gate applies symmetrically here — two near-simultaneous rejection submissions for the same order could both pass the idempotency check before either write commits. The worst case is one extra (harmless, idempotent) inventory-release/coupon-release call plus a possible duplicate `payment_events` row and a possible duplicate rejection email — not a money-losing or data-corrupting outcome. The optional partial-unique-index hardening B1 already proposed and deferred (`uq_payment_events_bank_transfer_confirmed`) has a natural, equally-optional B2 sibling (`uq_payment_events_bank_transfer_rejected`, or a single combined index on `(order_id, event_type)` covering both) — see §8.

## 6. Already-paid protection — detailed behavior

**Trigger condition:** an admin attempts to run `mark_bank_transfer_not_received` (or, if extended per §3 item 3's open question, `cancel_order`) against a bank-transfer order whose `payment_status` is already `'paid'` (via B1's approval helper, or any other path) or whose `status` is already in a fulfillment-advanced or settled state (`'preparing','shipped','delivered','refunded','partially_refunded'`).

**Expected behavior, two layers deep (defense in depth, both already exist or are added by this plan — neither replaces the other):**
1. **Outer layer (existing, unchanged):** both `admin/orders.js` (line 335-337) and `admin/orders/[id]/status.js` (line 38-40) already return an HTTP 409 with a clear Turkish error (`'Ödemesi alınmış sipariş doğrudan iptal edilemez. Önce kontrollü iade sürecini başlatın.'` / `'...Kontrollü iade akışını kullanın.'`) **before** reaching the generic status-mutation block at all — so in the two production call sites this plan touches, the new helper is never actually invoked for an already-paid order under normal operation.
2. **Inner layer (new, this plan):** `rejectManualBankTransferPayment()` itself independently re-checks the order's `payment_status`/`status` and refuses to proceed if already paid/settled, returning a structured `{ok:false, blocked:true}` result rather than throwing — so that if the helper is ever called from a future third call site that forgets to replicate the outer 409 guard (or if a caller's guard has a bug), the helper itself is still the last line of defense against the specific, most dangerous failure modes:
   - **Do not mark a paid order's `payments` row `'failed'`** — the guard fires before any `payments` write.
   - **Do not release inventory that has already been converted** (i.e., already deducted from `stock_on_hand` and no longer sitting in `stock_reserved`) — the guard fires before the `releaseInventoryReservations()` call; note `release_order_inventory`'s own reserved-status matching would likely already no-op safely here regardless (same idempotency property relied on elsewhere in this plan), but the explicit guard makes the *intent* — "never even attempt this for a paid order" — unambiguous in the code, rather than relying on an RPC-level side effect to save it.
   - **Do not release an already-used coupon** — the guard fires before any `coupon_redemptions` write, so a coupon already finalized to `'used'` by B1's approval helper is never reverted to `'released'` by a subsequent, mistaken rejection attempt.
   - **Do not reverse loyalty points via this helper** — `rejectManualBankTransferPayment()` never calls `reverseOrderPoints()` at all (that hook lives, unchanged, in the generic before/after-transition block in both calling routes, gated on the order's `isSettled`/`wasSettled` transition, not on this helper) — so there is no risk of this helper introducing an incorrect reversal path. If a genuine refund/cancellation-after-payment flow is needed, that is explicitly a different, future, controlled-refund batch (out of scope here, consistent with the existing 409 guard's own error message directing the admin to "the controlled refund process").

**Response shape for the blocked case:** the calling routes should surface this as the same 409 they already return today (unreachable in practice, per layer 1 above) — this plan does not propose changing either route's HTTP-level response for this case, only adding the belt-and-suspenders internal guard described above.
